  1. 环境准备
================================================================================

操作系统: Linux (Ubuntu 22.04+ 或兼容发行版)
编译器:   Rust 1.75+
依赖:     libc (系统自带)

步骤:
  $ git clone https://github.com/liuhuaxu463/MiniOS.git
  $ cd MiniOS

================================================================================
  2. 编译项目
================================================================================

命令:
  $ cargo build --release

预期结果: 编译成功，生成 target/release/minios 可执行文件

验证:
  $ ls -lh target/release/minios
  $ ./target/release/minios --help

预期输出: 显示帮助信息，列出所有命令和参数

## 重要提示：为了方便测试，建议先设置一个别名或变量

  # 方法1: 使用 shell 变量
  $ MINIOS=./target/release/minios
  $ $MINIOS --help

  # 方法2: 创建软链接
  $ sudo ln -s $(pwd)/target/release/minios /usr/local/bin/minios

  # 方法3: 直接使用完整路径（本文档所有示例均使用此方式）
  $ ./target/release/minios --help

  3. 服务器启动
================================================================================

命令 (前台运行):
  $ ./target/release/minios --server \
      --socket-path /tmp/minios_test.sock \
      --store-path ./test_store.odb \
      --log-level info

预期结果:
  - 终端输出 "MiniOS server is ready"
  - 生成 /tmp/minios_test.sock 文件
  - 生成 ./test_store.odb 文件

验证:
  $ ./target/release/minios --socket-path /tmp/minios_test.sock status

预期输出: 显示服务器状态面板，包含 Storage、Cache、Shared Memory 三项

  字段               预期值
  ──────────────────────────────────────
  Objects:            0 / 1000
  Capacity:           0 / ~200MB
  Data blocks:        已计算
  Cache entries:      0 / 128
  Hit rate:           0.00% (无操作时)
  Shared Memory:      所有页空闲

  4. 对象 CRUD 操作测试
================================================================================

----------------------------------------------------------------------
  4.1 PUT: 上传小文本文件
----------------------------------------------------------------------

准备:
  $ echo "Hello, MiniOS! This is a test file." > /tmp/test_small.txt
  $ echo "第二行内容" >> /tmp/test_small.txt

命令:
  $ ./target/release/minios --socket-path /tmp/minios_test.sock put \
      --name "test-small" \
      --file /tmp/test_small.txt \
      --type "text/plain" \
      --tags '{"author":"tester","level":"1"}'

预期输出:
  ✓ Object stored successfully!
    UUID:        xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
    Name:        test-small
    Size:        52 (52 B)
    Type:        text/plain
    Created:     2026-06-01 xx:xx:xx
    Tags:        {"author":"tester","level":"1"}
    Data blocks: 1

----------------------------------------------------------------------
  4.2 PUT: 上传大文件 (测试多块存储)
----------------------------------------------------------------------

准备:
  $ dd if=/dev/urandom of=/tmp/test_large.bin bs=1024 count=100

  生成 100KB 随机文件 (需要 >4KB 以跨多个存储块)

命令:
  $ ./target/release/minios --socket-path /tmp/minios_test.sock put \
      --name "test-large" \
      --file /tmp/test_large.bin \
      --type "application/octet-stream" \
      --tags '{"size":"large"}'

预期输出:
  ✓ Object stored successfully!
    Data blocks: 25 (100KB / 4KB = 25 blocks)

  5. 缓存功能验证
================================================================================

测试流程:

  1. 查看初始缓存状态:
     $ minios ... status
     预期: Hits: 0, Misses: 0, Hit rate: 0.00%

  2. 首次 GET test-small (缓存未命中):
     $ minios ... get --key "test-small" --output /dev/null
     预期: 正常下载

  3. 查看状态 (应有 1 次 miss):
     $ minios ... status | grep -A5 "Cache"
     预期: Hits: 0, Misses: 1

  4. 再次 GET test-small (缓存命中):
     $ minios ... get --key "test-small" --output /dev/null
     预期: 正常下载 (从缓存读取，更快)

  5. 查看状态 (应有 1 次 hit, 1 次 miss):
     $ minios ... status | grep -A5 "Cache"
     预期: Hits: 1, Misses: 1, Hit rate: 50.00%

  6. 多次 GET 同一对象:
     重复步骤 4-5 若干次
     预期: 命中率逐渐上升

================================================================================
  6. 数据持久性验证
================================================================================

测试流程:

  1. 正常停止服务器:
     $ ./target/release/minios --socket-path /tmp/minios_test.sock stop

  2. 验证 socket 和 PID 文件已清理:
     $ ls /tmp/minios_test.sock     → 应该不存在
     $ ls /tmp/minios_test.pid      → 应该不存在

  3. 验证 store.odb 存在:
     $ ls -lh ./test_store.odb      → 文件存在，大小 > 4KB

  4. 重新启动服务器:
     $ ./target/release/minios --server \
         --socket-path /tmp/minios_test.sock \
         --store-path ./test_store.odb &

  5. 验证数据依然存在:
     $ ./target/release/minios --socket-path /tmp/minios_test.sock list
     预期: test-small 仍在列表中

  6. 下载并验证内容:
     $ ./target/release/minios --socket-path /tmp/minios_test.sock get \
         --key "test-small" --output /tmp/after_restart.txt
     $ diff /tmp/test_small.txt /tmp/after_restart.txt
     预期: 无差异

  9.2 运行时动态缩放缓存

预期输出:
  ✓ Cache resized from 32 to 512

----------------------------------------------------------------------
  9.3 缓存算法对比 Benchmark
----------------------------------------------------------------------

前置条件: 服务器中有至少 3-5 个对象（可以先上传一些不同大小的文件）

命令:
  $ ./target/release/minios --socket-path /tmp/minios_test.sock cache-benchmark --iterations 200

预期输出:
  Workload:  N unique objects, 200 iterations

  Rank    Algorithm    Hits         Misses       Hit Rate
  ------+------------+------------+------------+----------
  🥇 1st  LFU          xxx          xxx          xx.xx%
  🥈 2nd  LRU          xxx          xxx          xx.xx%
  🥉 3rd  FIFO         xxx          xxx          xx.xx%

  Best algorithm: XXX (xx.xx% hit rate)

说明:
  - 三种算法使用相同的预加载数据和相同的工作负载
  - 排名根据命中率从高到低排列
  - 如果没有上传任何对象，benchmark 会返回 "No objects stored" 错误
  - 此结果可用于课程报告中分析"不同 workload 下哪种算法最优"
